You are building the data aggregation layer for Running.Services.

Product context:
Running.Services is a race discovery and decision engine, not a generic race calendar. The goal is to help runners choose the right race by comparing 5Ks, 10Ks, half marathons, marathons, trail races, Turkey Trots, charity runs, and fun runs by date, location, distance, price, registration urgency, terrain, difficulty, vibe, family-friendliness, beginner-friendliness, PR potential, and organizer trust.

Build a source aggregation system around 7 source types and 8 priority platforms.

7 source types to support:
1. Registration platforms
2. Official race websites
3. Organizer / claim submissions
4. Timing and results providers
5. Course / map sources
6. Local running institutions
7. Context and enrichment APIs

8 actual platforms/sources to prioritize:
1. RunSignup
2. Race Roster
3. ACTIVE
4. Eventbrite
5. Athlinks / ChronoTrack ecosystem
6. RunGuides
7. RaceRaves
8. RRCA club and event ecosystem

Important source strategy:
RunGuides and RaceRaves are useful for coverage gaps, reviews, and validation, but they should not be treated as canonical sources for core race facts. Canonical race facts should come from organizer-verified updates, official race websites, registration platforms, and timing/results providers.

Source trust hierarchy:
1. Organizer-verified claim/update: highest trust
2. Official race website: very high trust
3. Registration platform: very high trust
4. Timing/results provider: high trust
5. RRCA/local club calendar: medium trust
6. Eventbrite/social event page: medium-low trust
7. Third-party race directory: low trust for canonical facts, useful for discovery

Primary data objective:
Create a normalized race-intelligence database that combines:
- registration listings
- official race details
- organizer-verified updates
- historical results
- course difficulty
- price/deadline tracking
- local club calendars
- weather/location intelligence

Deduplication:
Implement race deduplication before writing canonical race records.

Deduping signals:
- normalized race name similarity
- date proximity
- city/state match
- venue match
- distance overlap
- organizer match
- registration URL match
- official website URL match
- source platform external ID
- geolocation proximity

Deduping logic:
- If registration URL or official website URL matches, treat as high-confidence duplicate.
- If name + date + city/state match, treat as likely duplicate.
- If name similarity is high and date is within 1-3 days and location is nearby, flag for review.
- Keep all source records attached to one canonical Race record.
- Never discard raw source records.
- Store source provenance for every field.

Field-level conflict resolution:
When multiple sources disagree, resolve fields using the trust hierarchy:
1. Organizer-verified claim/update
2. Official race website
3. Registration platform
4. Timing/results provider
5. RRCA/local club calendar
6. Eventbrite/social event page
7. Third-party race directory

For each canonical field, store:
- value
- source_url
- source_platform
- confidence
- last_verified_at

Verification and freshness:
Add freshness rules:
- Race date, registration URL, and location should be reverified frequently for upcoming races.
- Price and deadline fields should be rechecked more often as race date approaches.
- Past race results should be checked after race date.
- Pages without recent verification should show a “last verified” date.
- Do not index stale or incomplete race pages.

Indexing rules:
A race page may be indexable only if it has:
- name
- date
- city/state
- at least one distance
- registration URL or official website URL
- source URL
- last verified date

Noindex:
- empty pages
- thin pages
- duplicate pages
- stale pages without verified current date
- city/distance pages with fewer than 5 upcoming races unless they include nearby alternatives and useful editorial context
